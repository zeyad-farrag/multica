## Approach (this repo)

1. **Author `team-app/migrations/001_init.up.sql`** wrapped in a single `BEGIN; ... COMMIT;`:
   - Create the 4 mirror tables in order — `mirror_workspace`, `mirror_user`, `mirror_member`, `mirror_issue` — exactly per the schema reference in the issue (mirror tables are workspace-keyed in both 001 and 002; no reshape).
     - `mirror_workspace.work_week JSONB NOT NULL DEFAULT '{}'::jsonb` — column exists from day 1; behaviour enforcement (no writes from WS subscriber/reconciler) is a later-story concern.
     - Add the four mirror indexes per the schema reference (`idx_mirror_member_workspace`, `idx_mirror_member_user`, `idx_mirror_issue_workspace`, `idx_mirror_issue_assignee`, `idx_mirror_issue_parent`).
     - Mirror tables MUST NOT carry FOREIGN KEYs to Multica tables (Multica lives in a separate database — architecture §149-§157, §771-§775). Document the relationship in SQL comments only.
   - Create the 8 domain tables in dependency order — `member_schedule`, `member_leave`, `work_item`, `time_entry`, `time_confirm`, `time_confirm_history`, `workload_anomaly`, `activity_log` — using v2 column names: `member_id` (NOT `user_id`), `assignee_id`/`creator_id` (no `_user_id`), `confirmed_by`/`created_by` (no `_user_id`). NO `organization_id` columns. NO FK to `organization` (table doesn't exist yet).
     - `time_confirm` ships in v2 shape only (no `status`, no `employee_comment`/`manager_comment`/`manager_user_id`/`submitted_at`/`reviewed_at`/`withdrawn_at`). Columns per Dev Notes guardrail #4. UNIQUE on `(workspace_id, member_id, week_start)`.
     - `time_confirm_history` ships as a plain table — NO append-only trigger in 001 (the trigger lands in 002 per spec §6.2). Schema columns per spec §5.0.5.
     - `time_entry` and `work_item` carry the 15-min DB CHECKs (`minutes % 15 = 0 AND > 0`, `duration_minutes % 15 = 0 AND > 0`) per AC-3 (NFR26, NFR27).
     - `time_entry` carries the XOR check across (`kind`, `issue_id`, `work_item_id`, `work_item_title_snapshot`) and the confirmed-row CHECK (`confirmed=FALSE OR (confirmed_at IS NOT NULL AND locked_at IS NOT NULL)`) and `kind <> 'other' OR source = 'manual'`.
     - `member_schedule` carries `CHECK (work_end > work_start)`, `CHECK (array_length(work_days,1) BETWEEN 1 AND 7)`, `CHECK (work_days <@ ARRAY[0..6])`.
     - `member_leave` carries `CHECK (ends_on >= starts_on)`.
     - `work_item` carries the recurrence-parent/instance_date XOR check and the `cancelled_override XOR recurrence` check; recurrence-parent FK is self-referential.
     - `time_entry.work_item_id` FK → `work_item(id) ON DELETE SET NULL` (only intra-domain FK in 001).
     - `time_confirm_history.time_confirm_id` FK → `time_confirm(id) ON DELETE CASCADE`.
   - Create per-table non-unique support indexes per the schema reference (e.g., `idx_member_leave_range`, `idx_work_item_workspace`, `idx_work_item_assignee_time`, `idx_work_item_recurrence_parent`, `idx_work_item_override_unique`, `idx_time_entry_member_started`, `idx_time_entry_issue`, `idx_time_entry_work_item`, `idx_time_entry_unconfirmed`, `idx_time_entry_conflict_group`, `idx_time_confirm_history_confirm`, `idx_workload_anomaly_workspace`, `idx_workload_anomaly_member`, `idx_activity_log_workspace`, `idx_activity_log_actor`).
   - Create the partial unique index `idx_time_entry_auto_issue_key` on `(member_id, issue_id, started_at)` with predicate `kind='issue' AND source='auto' AND confirmed=FALSE AND user_edited=FALSE` (AC-4 — uses `member_id` because the rename to `user_id` happens in 002).
   - Create the partial unique index `idx_workload_anomaly_open` on `(member_id, kind, window_start, window_end)` with predicate `resolved_at IS NULL` (AC-5 — uses `member_id`; the rename + `organization_id` extension happen in 002).
2. **Author `team-app/migrations/001_init.down.sql`** wrapped in `BEGIN; ... COMMIT;` with `DROP TABLE IF EXISTS` in this exact reverse-creation order: `activity_log`, `workload_anomaly`, `time_confirm_history`, `time_confirm`, `time_entry`, `work_item`, `member_leave`, `member_schedule`, `mirror_issue`, `mirror_member`, `mirror_user`, `mirror_workspace`. Indexes drop with their tables — no separate `DROP INDEX`.
3. **Author `team-app/sqlc.yaml`** with `version: "2"`, one `sql` block: `engine: postgresql`, `queries: "pkg/db/queries"`, `schema: "migrations"`, `gen.go.package: "generated"`, `gen.go.out: "pkg/db/generated"`, `gen.go.sql_package: "pgx/v5"`, `gen.go.overrides`: UUID→`github.com/google/uuid.UUID`, JSONB→`[]byte` (or `json.RawMessage` if project-wide convention prefers — pick one and document in the file).

Each file contains at least one named `:one` SELECT per table in its family — enough to force sqlc to compile every column type. 7. **Author `team-app/migrations/migration_test.go`** (or `team-app/internal/db/migration_test.go` if Story 1.1 placed migration code under `internal/db/`):

   - Test asserts both partial unique indexes exist via `pg_indexes` and that their `indexdef` contains the predicate substring (`kind = 'issue'` for the time_entry one, `resolved_at IS NULL` for the workload_anomaly one).
   - Test asserts the 15-min CHECKs by attempting `INSERT ... minutes = 10` (expect failure), then `INSERT ... minutes = 15` (expect success), and the equivalent for `work_item.duration_minutes`.
   - Test then applies `001_init.down.sql` and asserts none of the 12 tables remain in `pg_tables` for the public schema (or whatever schema the standalone app uses).
8. **Verify locally**: `make sqlc` returns clean and `pkg/db/generated/` contains a `Queries` struct with methods for every named query; `go build ./...` and `go vet ./...` pass; the migration test passes against a fresh Postgres 17 container.

## Test strategy (this repo)

- **Unit / Integration**: `team-app/migrations/migration_test.go` (or `team-app/internal/db/migration_test.go`) — real Postgres (pgvector/pg17), no mocks. Covers AC-1 (12 tables present), AC-3 (15-min CHECK enforcement), AC-4 / AC-5 (partial unique index predicates), AC-6 (down strip).
- **Build-time**: `make sqlc` regen + `go build ./...` + `go vet ./...` covers AC-7 (sqlc compiles against the schema).
- **Manual verification**: run `migrate up && migrate down && \dt` against a local empty Postgres and confirm zero tables — backstop for the AC-6 strict-inverse claim. (The Go test asserts the same thing programmatically; the manual check is for human sanity on first land.)

Story 1.6 is the **pre-org, v2-keyed** shape only.
- Production queries against the new tables — stubs only here. WS subscriber queries land in Story 1.7; reconciler queries in Story 1.8.
- Mirror-population logic / WS subscriber / reconciler — schema only; no code that writes mirror rows.
- Enforcement of "WS subscriber and reconciler MUST NEVER write `mirror_workspace.work_week`" — that is a query/handler concern in later stories. This story creates the column with its default; behaviour rules live in code that does not exist yet.
- Adding FKs from mirror tables to Multica — forbidden (cross-database FK; architecture §149-§157, §771-§775). Mirror UUIDs equal Multica UUIDs by convention only.
